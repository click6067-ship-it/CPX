## 0. 한눈에 보기 (요약)

CPX-AI는 **의대 임상수행평가(CPX) 교육을 돕는 AI 시스템**입니다. 한 문장으로 하면 **"CPX 사례 개발과 연습을 *보조*하는 연구용 AI 시뮬레이터"** 입니다(교수를 대체하지 않고 돕습니다). 시험 사례를 **쓰는 AI(①)**, 그것을 **검수하는 AI(②)**, 환자를 **연기하는 AI(③)**, 학생을 **채점하는 AI(④)** 네 명과, 이들을 받치는 뒷단(도서관·통역사·품질검사실)으로 이루어집니다.

현재는 **②(검수 AI)가 교수의 피드백을 얼마나 재현하는지**를 교수 블라인드 설문으로 검증하는 **파일럿 단계**입니다.

| 항목 | 내용 |
|---|---|
| **무엇** | CPX 사례 생성(①)·검토(②)·가상환자(③)·자동채점(④) + 검증 하네스 |
| **누가/왜** | 양산부산대병원 연구책임자 연구. 부산대 학생 무료 제공·오픈소스·논문 목적 |
| **현재 단계** | ②검토 AI의 H2 검증 파일럿(웹 설문 배포·교수 평가 수집 중) |
| **핵심 원칙** | 투명성·재현성, **과대주장 금지**("AI=전문가 동등" 주장 안 함), 데이터 누수 방지 |
| **모델** | ①②=gpt-5.5, ③④=GPT-4o(미구현), RAG 임베딩=Gemini, 검색=BM25 |
| **솔직한 한계** | feasibility 파일럿(소표본). 타당성(통계적 동등) 주장 아님 |

> CPX(Clinical Performance Examination, 임상수행평가) = 의대생이 **표준화 환자(연기자)**를 직접 문진·진찰하고, 그 수행을 체크리스트로 평가하는 실기시험. 국가고시 항목.

## 1. 배경과 맥락 — 왜 이 프로젝트인가

### 1.1 문제
CPX 교육에는 두 가지 비용이 큽니다.
1. **사례 제작 부담** — 좋은 CPX 사례 하나를 만들려면 교수가 시나리오·체크리스트·환자 답변·채점기준을 모두 설계하고, 동료 검토를 거쳐야 합니다. 시간이 많이 듭니다.
2. **연습 기회 부족** — 학생이 표준화 환자와 연습하려면 사람(연기자)·공간·채점자가 필요해, 반복 연습이 어렵습니다.

### 1.2 접근
AI로 이 과정을 **보조**합니다. 기존 사례를 변형해 새 사례를 **생성(①)**하고, AI가 **검토(②)**해 교수 확정을 돕고, 확정된 사례로 **가상환자(③)**가 학생과 대화하고, **자동채점(④)**으로 즉시 피드백을 줍니다.

### 1.3 성격 (오해 방지 — 가장 중요)
- **학술 연구 프로젝트**입니다. 영리·사업·스타트업이 **아닙니다.**
- 목적은 **논문 + 검증(validation) + 투명성**입니다. 시장·가격·마케팅과 무관합니다.
- 산출물은 부산대 학생에게 **무료**, 작동방식은 **전면 공개(오픈소스)**, 결과로 **논문**이 따라옵니다(논문은 교수·연구진 주도, 개발 담당는 학부 개발 담당).
- 따라서 평가·비평은 **학술 엄밀성·검증 타당성·재현성·의학적 안전성·과대주장 방지** 축으로만 합니다.

### 1.4 개발·검토 방식
모든 핵심 작업은 **두 AI의 교차 검증**을 거칩니다. Claude가 설계·구현하고, **Codex(다른 AI 모델)가 적대적으로 검수**(red-team)합니다. 이 보고서 자체도 Codex 검수를 받습니다. 주요 작업은 `worklog.md`에 기록됩니다.

## 2. 전체 구조 (아키텍처 한 장)

![CPX-AI 작동방식](cpx-flow.png)

시스템은 **세 영역**으로 나뉩니다.

```
A. 사례 개발 루프 (교수 보조)        B. 검증 (현재 파일럿)         C. 학생 런타임 (목표)
  씨앗 사례                            교수 과거 피드백              확정 사례
    │                                      │ + ② AI 리뷰               │
    ▼                                      ▼                          ▼
  ① 생성 ──▶ ② 검토 ──▶ 확정?           블라인드 설문               ③ 가상환자
    ▲           │  (RAG 근거)            (교수 A/B 익명)             (GPT-4o)
    └─ 수정 ◀───┘                          ▼                          ▼
                ▼                        집계                       학생 대화
            확정 사례                  (recall·정밀도·ICC)            ▼
                                                                 ④ 자동채점 ─▶ 피드백
```

**등장 인물(비유로 이해하기):**

| 모듈 | 비유 | 하는 일 |
|---|---|---|
| ① `generator` | ✍️ 사례 작가 | 증상+진단(또는 기존 사례)으로 새 사례 초안 작성 |
| ② `reviewer` | 🔍 심사위원 | 점검표로 구조+임상 결함을 지적, 수정안 제시 (**현재 검증 대상**) |
| ③ `patient` | 🎭 환자 배우 | 사례 속 환자를 연기, 묻는 것만 답변(과공개 제어) |
| ④ `grader` | ✅ 채점관 | 학생 대화를 체크리스트와 대조해 점수·피드백 |
| `rag` | 📚 도서관 사서 | 질문에 맞는 의학 교과서 근거를 찾아 ②에게 제공 |
| `llm` | 🔌 통역사(어댑터) | 모든 AI 호출 창구. 모델 교체는 이 파일 한 곳만 수정 |
| `harness` | 🔬 품질검사실 | "이 채점·검토를 믿어도 되나"를 정답지와 대조해 숫자로 검증 |
| `graph` | 🏭 관리자(LangGraph) | 생성→검토→수정→재검토 자동 루프 관리 |

핵심 동사 3개로 기억: **만든다(①②) · 연습한다(③④) · 믿는다(검증 하네스).**

## 3. 작동 원리 (단계별 상세)

### 3.1 A. 사례 개발 루프

1. **입력**: 증상 + 진단, 또는 기존 CPX 사례(씨앗).
2. **① 생성**: AI 작가가 도서관(RAG)의 교과서 근거를 참고하며 사례 초안을 작성합니다. "65세 남성 흡연자"만 반복 생성되는 편향(→ §8)을 막기 위해 **인구통계 다양성을 유도**합니다. *(현재는 프롬프트 지시 수준이며, 진단별 분포 샘플러·검증기는 후속 과제입니다.)*
3. **② 검토**: 심사위원 AI가 두 층으로 점검합니다.
   - **②A 구조 리뷰어**: 체크리스트 항목 수, 복합질문 분리, 채점 명료성, 진찰·교육 항목 등 *형식·구조*.
   - **②B 임상 리뷰어**: 임상 타당성, 핵심 병력 누락, red flag, 위험인자, 내적 논리, 표준화 환자 현실성 등 *내용*. 이때 RAG 근거를 활용합니다.
   - 출력: 각 지적에 **카테고리·근거·수정안·심각도**를 붙입니다.
4. **확정 판단**: Accept / Minor / Major / Reject. 미흡하면 ①로 돌아가 수정 후 재검토(이 루프를 LangGraph가 자동화). **최종 확정은 교수(HITL, Human-in-the-loop)** 가 합니다.

### 3.2 B. 검증 (H2, 현재 파일럿)

"②의 AI 검토가 교수 피드백을 얼마나 재현하는가"를 측정합니다. 자세한 설계는 §7.

### 3.3 C. 학생 런타임 (목표 산출물)

1. **③ 가상환자**: 확정 사례로 환자를 연기합니다. **항목별 답변 정책** — 허용 사실, 트리거 표현, 거부·되묻기 규칙, **과공개 억제**(묻지도 않은 정보를 흘리지 않게). *(완전한 유한상태기계(FSM)는 설계안이며, 현재 구현은 프롬프트 기반입니다.)* 텍스트 우선, 음성(STT/TTS)은 옵션·후반.
2. **학생 대화**: 학생이 문진·교육을 진행, 대화록이 남습니다.
3. **④ 자동채점**: 대화록을 체크리스트와 대조합니다.
   - **병력청취·신체진찰 = 이분법**(도달/누락). 복합질문은 "또는"으로 분리.
   - **환자의사관계(PPI, 20%) = 이분법 부적합** → 루브릭 + 인간 보조.
   - 뜻이 같으면 표현이 달라도 인정(의미 기반). 키워드 방식(v0)도 병존해 비교.
4. **디브리핑(첨삭)**: 발화별 유도성 질문·전문용어·개방형/폐쇄형 여부 + "이렇게 말하라"는 리라이트 제안.

### 4.1 데이터 누수 방지 (연구 무효화 1순위 함정)
같은 사례를 *생성 예시 + 튜닝 + 가상환자 조정 + 최종평가*에 동시에 쓰면 결과가 조용히 무효가 됩니다. 그래서 데이터를 **train(생성용) / dev(튜닝용) / locked(평가 전용)** 로 **미리** 분리하고, locked는 평가 당일까지 열지 않습니다. 생성 사례와 실제 사례는 provenance 태그로 구분합니다. *(분할 manifest·해시·각 split 개수는 `data/working/`에 기록되어 "locked를 안 열었다"를 사후 검증 가능.)*

### 4.2 보안·저작권
- 학교자산(사례·피드백)과 저작권 교과서는 **git 커밋 금지.**
- 외부 AI API에는 **비식별 데이터만** 입력하고, **재학습(파인튜닝)하지 않습니다**(프롬프트 + RAG 방식). 외부 API는 **no-training(학습 미사용) 설정**으로 사용합니다(OpenAI·Google).
- 키는 `.env`(비공개). 배포 설문은 비식별 항목만, 암호 게이트 + 검색엔진 차단.

| ③ 가상환자 · ④ 자동채점 | GPT-4o | OpenAI | 미구현 | 계획서 지정 |
| RAG 임베딩(dense) | gemini-embedding-001 | Google | 사용 중 | 다국어(한↔영). 생성 LLM이 아니라 글→벡터 변환 전용 |
| RAG 검색(sparse) | BM25 (rank_bm25) | — | 사용 중 | 정확 용어 매칭. 로컬 계산, API·모델 없음 |
| 보조·사례변환(ingest) | gemini-2.5-flash | Google | 사용 중 | 빠름·저렴. ②리뷰와 분리 |
| 적대 검수 | Codex | OpenAI/Codex | 사용 중 | 설계·코드 red-team(메타만 전송) |

### 5.1 어댑터 설계 (핵심)
모든 모델 호출은 `src/cpx/llm.py` **한 파일**을 거칩니다. 모델명 앞부분으로 Claude/GPT/Gemini를 자동 구분합니다. 덕분에 **싼 모델로 검증 → 좋은 모델로 한 줄 교체**가 가능하고, Claude 결제가 풀리면 ①②를 Claude로 즉시 되돌릴 수 있습니다.

> **검증 대상 모델 고정:** ② 모델은 **배포 설정·데이터 생성 로그 기준 `gpt-5.5`**로 고정하며, `transparency.md`·설문 배포 설정과 일치시킵니다(이 셋이 어긋나면 검증 대상이 흔들리므로 단일화).
>
> **솔직한 비교(계획서 vs 현재):** 계획서는 Claude=①②, GPT-4o=③④였습니다. Claude API 결제가 한국 카드 문제로 막혀, Claude 자리를 당시 사용 가능한 상위 GPT 모델(`gpt-5.5`)로 대체했습니다. 임베딩=Gemini는 다국어 때문에 우리가 선택했습니다(계획서에 명시 없던 부분).

## 6. RAG (검색 증강 생성) 작동방식

RAG = AI가 **"오픈북 시험"** 처럼 교과서 근거를 보고 검토하게 하는 구조입니다. 베끼지 않고 근거를 대게 합니다. (`src/cpx/rag.py`)

1. **색인(1회)**: 교과서를 조각(청크)으로 나눠 → 임베딩 API(Gemini)로 각 조각을 벡터로 변환 → **로컬 파일**(.npy)에 저장. *(현재 인덱스 = Harrison 1권 1,200청크 샘플. 18권 전체는 코퍼스 후보이며 아직 색인 전입니다.)*
2. **검색(질의 시, 하이브리드)**:
   - **dense**: 질문을 벡터로 바꿔 의미가 비슷한 조각을 코사인 유사도로 찾음(한국어 질문 → 영어 책도 가능).
   - **sparse**: BM25로 정확한 용어가 일치하는 조각을 찾음(로컬 계산).
   - **RRF 병합**: 두 결과의 순위를 합쳐 최종 top-k.
3. **주입+생성**: top-k 근거를 ② 프롬프트에 넣어, gpt-5.5가 근거 기반으로 검토.

- **외부 API 호출 = 색인 시 청크 임베딩(1회) + 검색 시 질의 임베딩(매 질의) + 생성 LLM 호출.** 벡터 저장은 파일, BM25·병합은 메모리 → **벡터DB 서버가 필요 없습니다**(코퍼스가 작아서). 수백만 청크 규모면 벡터DB가 필요하지만, 우리 규모는 아닙니다.
- **현재 한계(중요)**: 프로젝트 원칙은 *한국 CPX 자료를 주 코퍼스*로 두는 것이지만, **현재 색인된 것은 영어 교과서(Harrison 샘플)뿐**입니다. 한국어 질의는 dense(다국어)가 주력이며, 한국 CPX 총론·지침·점검표 색인과 저작권 처리는 **후속 과제(gap)**입니다.

## 7. 검증(H2) 설계 — 핵심

> **H2 가설**: "AI ②검토자가 전문가(교수) 피드백의 어떤 유형을 어느 정도 재현하는가?" *(동등/필적 주장이 아니라, 재현 정도와 차이를 탐색적으로 측정.)*

### 7.1 왜 설계가 까다로운가 (v1의 결함, Codex가 깸)
처음 설계(v1)는 "AI가 교수 지적을 쪼개고 → AI가 매칭하고 → 교수가 맞나 확인"이었습니다. Codex가 다음을 지적해 **REVISE**:
1. **순환 측정** — AI가 만든 후보 안에서만 교수가 판정 → 결과가 모델 산물인지 교수 판정인지 분리 안 됨.
2. **앵커링** — AI 답을 먼저 보여주면 교수가 끌려감(인지 편향).
3. **정보 부족** — 1줄 지적으로는 판정 불가 → 평가자 간 신뢰도 붕괴.
4. **약한 주지표** — 항목별 recall은 부차적. 주지표는 사례 단위 비교여야.
5. **정밀도 누락** — AI가 새로 낸 지적의 타당·유해성 평가 필요.

### 7.2 v2/v3 재설계 (현재 배포본)
**판정 단위 = 사례. 한 사례 = 화면에 전부 표시**(초안 전문 + 두 리뷰 전문 + AI 리뷰 전체).

- **1단계 [주지표] 블라인드 루브릭** — 같은 초안의 두 리뷰(하나는 교수 피드백, 하나는 AI)를 **무작위·익명**으로 제시. 교수가 어느 쪽이 AI인지 모른 채 **완전성·정확성·유용성·안전성(1~5)** 로 평가. → AI 리뷰와 전문가 피드백의 **탐색적 유사도·차이**를 측정(앵커링 제거). *(현재는 동등성/비열등성을 주장하지 않으며, 비열등성 margin·표본수·primary endpoint는 본검증에서 사전등록합니다.)* 단, 같은 교수가 *자신이 과거에 쓴* 피드백을 평가할 경우 완전한 블라인드가 아닐 수 있어, 평가자별 **원 피드백 작성자 여부**를 기록해 분석에서 분리합니다.
- **2단계 [보조] recall** — 어느 쪽이 AI였는지 공개 후, AI 리뷰 전체를 보며 교수 지적 각각을 caught/partial/missed/excluded.
- **3단계 정밀도** — AI가 낸 지적 각각: 타당·중요 / 타당·경미 / 중복 / 틀림 / **위험·유해**.

**블라인드가 깨질 위험**(AI와 교수의 문체·길이 차이로 들킴)을 막기 위해 v3에서:
- 두 의견을 **같은 형식(불릿)으로 정규화** + 순서 무작위.
- **"어느 쪽이 AI 같나 + 확신도"** 를 물어 **블라인드 성공률**을 점검(50% 근처면 양호).
- 각 1~5 척도에 **BARS 앵커**(1/3/5 기준 서술) + 전문 **읽음 확인** 체크.
- **모드 분리**(`?mode=blind`/`recall`/`full`)로 평가자 코호트를 나눠 단계 간 오염 방지.

### 7.3 지표
- (1) 차원별 **AI − 전문가 차이 + 사례 클러스터 부트스트랩 95% CI**, **ICC(2,k)**.
- (2) recall(다수결 합의) + **Fleiss kappa**(현재 구현) + 카테고리별. *(weighted/Gwet kappa는 본검증 예정.)*
- (3) 정밀도 / false positive / **harmful = 안전 게이트**: 사전 정지규칙으로 운용(예: harmful 판정 1건 이상이면 릴리즈 보류).
- (4) 블라인드 성공률.
- 판정자 N명이 독립 평가 → 평가자 간 일치도 보고. **전문가-전문가 일치도(floor)를 먼저** 봅니다.

### 7.4 하이브리드 설문 (교수 부담 ↓ + 데이터 확보)
교수는 사례마다 **"블라인드 평가"(정량 데이터) 또는 "열람만"(자유 메모)** 를 고릅니다. 원하는 사례만, 언제든 종료 가능. 블라인드 무결성을 지키면서(평가를 라벨 보기 *전에* 함) 부담을 낮춥니다. 배포 = `web/adjudication/`(암호 게이트·중앙 수집·비식별).
> **분석 원칙(선택편향·결측 방지):** **블라인드 평가를 완료한 건만 정량 주지표**에 포함하고, **"열람만" 응답은 정성 메모로 별도 보고**합니다. 하이브리드 응답의 자동 집계는 데이터 수집 후 추가 예정이라, 그 전까지 하이브리드는 "권장"이 아니라 "선택지"로 안내합니다(현재 집계 스크립트는 full/blind/recall 지원).

### 7.5 파일럿 결과 (솔직하게)
- dev_tune 3건, **자동 LLM 판정**(소표본, locked 미사용): **② recall ≈ 23%**. ⚠️ 이 수치는 성능 측정이 아니라 **프롬프트 디버깅용 신호**입니다(아래 개선 수치도 동일 한계).
- **Codex 판정**: 이 23%는 **두 신호의 혼합** —
  1. *구성 불일치 아티팩트*: 당시 ②는 일반 구조 점검표였는데 교수 피드백은 사례특이 임상 정교화라 결이 달랐음(n=3, 자동 판정, 교수 피드백이 순수 정답지가 아님).
  2. *②의 진짜 한계*: 사례특이 임상 비평이 약함.
- **조치**: ② 2층 분리(②A 구조 + ②B 임상) + 실험 재설계(인간 adjudication). ②B 추가 후 19% → 26%(자동 판정·소표본이라 역시 **신호 수준**, 성능 주장 아님).
- **결론**: 본검증 = 교수 인간 판정 + 카테고리별 + ②B 반복 개선. **과대주장 금지.**

## 8. 학술적 근거와 선행연구

> 출처: `docs/research-llm-cpx-sota.md` (134개 주장 → 25개 적대검증 → **22 확인 / 3 폐기**). 아래는 **확인된 것만** 근거로 씁니다.
> ⚠️ **인용 주의(Codex 2R)**: 아래 수치(88%·31%·3.3%·90.5%)는 deep-research 인용으로, **정확 DOI·절·수치는 원문 재확인 후에만 논문 인용**(현재 일부 미확인). 디펜스 시엔 숫자 단정보다 **"관련 연구가 멀티에이전트·이분법·RAG의 우위를 *방향*으로 보고"** 수준으로 말할 것.

### 8.1 단계별 SOTA
- **① 생성**: 구조화된 **멀티에이전트 프롬프팅 > 단일 RAG**. 멀티에이전트로 사용가능 OSCE 88% vs 단일 RAG는 31% 수정 필요(Wang 2024, JMIR 10.2196/82116 — *영어 OSCE 사례·전문가 사용가능성 평가 기준이며, 한국 CPX와 과제·지표가 동일하진 않음*). **"checklist-as-script"**(체크리스트를 생성 골격으로) 기법 확인 → 우리가 `[붙임2]` 양식을 생성 골격으로 쓰는 근거.
- **② 검증**: 지배적 방법 = 다수 전문가 내용타당도. **순서 = 전문가 간 일치도 먼저 → AI-전문가**. 전문가 HITL 수용률 ~87%. 단 정밀 품질 ICC는 낮을 수 있음(0.18) → 이분법 kappa 위주. silver-label로 부트스트랩.
- **③ 가상환자**: 학술 근거가 **가장 빈약 = 우리 기여 기회**. 차별점 후보 = 응답 일관성·정보 과공개 제어.
- **④ 채점**: **이분법 채점이 강력**(외부 벤치마크 기준). 한 평가에서 오픈모델(qwen3-32b)이 90.50%로 GPT-4o와 동급 — 단 *그 벤치마크 과제 기준*이며 한국 CPX 대화 채점과 동일하지 않습니다. 우리 H4-real 검증 전에는 **비용 절감 후보**로만 봅니다. 우리 D1(이분법) 결정과 방향 일치.

### 8.2 우리 방법의 위치
멀티에이전트+checklist-seed, 하이브리드 RAG, HITL 2라운드 심사 = **SOTA 정렬(차용)**. 블라인드/엄밀 검증, 가상환자 일관성·과공개 제어 = **차별화·기여 포인트**.

### 8.3 알려진 함정과 대응
- **인구통계 편향**("65세 남성 흡연자" 과생성): 실측으로 남성 79~84% 과대, 소수집단 0~1.5% 과소(PMID 40665345). → **생성 단계에서 인구통계 다양성을 유도**(현재 프롬프트 지시; 진단별 분포 샘플러는 후속).
- **의학 hallucination**: 하이브리드 RAG + 구조화 self-check가 효과적이라는 **외부 연구 보고**(MedTutor, 오류 ~3.3%). 이는 *문헌 수치*이며 **우리 시스템의 오류율은 아직 측정 전**입니다(현재 RAG는 Harrison 샘플이라 더 보수적으로 봐야 함). → 우리 RAG 설계의 *방향* 근거.
- **지표**: 정밀 ICC는 낮을 수 있으니 **이분법 kappa·도메인별 weighted kappa·F1 + bootstrap CI**, 인간-인간 floor 먼저.

> ⚠️ **인용 금지(적대검증 탈락)**: "AI 생성 문항이 인간과 심리측정적으로 동등"(npj) 등은 검증에서 폐기됨 → **우리 입장은 "전문가 검토 후 사용가능"**, "AI=인간 동등"은 주장하지 않음.

## 9. 한계와 정직성 (과대주장 방지)

- 현재는 **엔지니어링 smoke + feasibility 파일럿** 단계입니다. **타당성(통계적 동등) 주장이 아닙니다.**
- H2 파일럿은 사례 6건 수준 — "AI가 전문가에 필적/대체한다"고 **주장하지 않습니다.**
- 인간-AI 산문 블라인드는 문체 차이로 완전하지 않습니다(문헌도 인정) → 한계로 명기.
- 정답지(교수 피드백) 분해가 아직 일부 LLM 보조 → 향후 인간 codebook 고정 권장.
- **교수 과거 피드백 ≠ 완전 gold** (Codex 지적): 역사적 비교대상일 뿐, 사례 결함을 빠짐없이 담았다는 보장 없음 → 파일럿 지표는 **"historical-feedback recall"**(과거 피드백 대비 일치)로 해석, 완전 gold는 본검증의 *2인 독립→합의 분해 SOP*(`validation-design.md`)로 만든다.
- **RAG 저작권 미해결**: 저작권 교과서 청크의 외부 LLM 전송은 프로덕션 전 *오픈라이선스 코퍼스 / 로컬 LLM / 비저작권 요약*으로 닫아야(M2·M9 참조).
- 사전등록(OSF)·비열등성 margin·locked 최종평가는 **본검증 단계**에서.
- ③④ 학생 런타임은 코어는 있으나 음성·통합·실검증(H4-real: 실제 학생 대화 + 임상교원 라벨)은 미완.
- **IRB/윤리**: 현재 학생 데이터는 **미수집**입니다. H2 교수 설문은 비식별·자발적이며 IRB 필요 여부/면제는 교수팀이 판단합니다. 향후 **학생 런타임(③④)은 IRB·동의·PIPA 절차가 필요**하며 이는 교수팀 영역입니다.

## 10. 로드맵 (남은 일)

> 목표 = 검증된·투명한·오픈소스 CPX AI(부산대 학생 무료) + 결과로 따라오는 논문. 빌드-주도.

**바로 다음 (게이트):** 교수 파일럿 실행 → 집계 → **첫 H2 실결과**. 이후:
1. 결과로 **②B 보강** → 생성→검토→수정 루프로 **① 생성 품질** 향상("개선봇이 핵심").
2. **④채점 검증(H4-real)** — 실제 학생 대화 + 임상교원 라벨 → AI-인간 일치도(논문 핵심).
3. **지표·재현성·오픈소스 패키징·투명성 산출물**(논문 재료).
4. **본검증** — 사례·판정자 확대, 사전등록(OSF), 비열등성 δ.

**작은 작업(코드):** 하이브리드 응답 집계 지원, F1 지표 추가, RAG 코퍼스 확대(18권 전체·한국어 색인), 음성 완성도.

## 🛡 핵심 메커니즘 — 어떻게·왜 작동하나 (디펜스 심화)

> "그냥 AI한테 시켰다"가 아니라, 각 부품이 *왜 이 방식이고 어떻게 작동*하는지. **코드가 아니라 원리.** 교수 추궁 대비 — 각 항목의 "디펜스 한 줄"을 본인 말로 설명할 수 있어야 함.

### M1. LLM 어댑터 — 왜 모델을 갈아끼우나
- **무엇**: 모든 AI 호출이 `llm.py` 한 창구를 거치고, 모델명 앞부분(claude/gpt/gemini)으로 자동 분기.
- **원리**: 회사마다 API 호출 형식이 다름 → 공통 인터페이스로 감싸 "통역". 부르는 쪽은 모델이 누군지 몰라도 됨.
- **왜**: 싼 모델로 로직 검증 → 좋은 모델로 한 줄 교체. 결제 막힌 Claude를 gpt-5.5로 *즉시* 대체할 수 있던 이유.
- **디펜스 한 줄**: "모델 종속을 없애 검증·교체·재현이 자유롭습니다."

### M2. RAG 하이브리드 — AI가 교과서 근거를 어떻게 찾나
- **무엇**: ②검토·①생성 시 의학 교과서에서 관련 근거를 찾아 프롬프트에 넣음(오픈북 시험).
- **원리(두 검색을 융합)**:
  - **dense(임베딩)**: 문장을 *의미 벡터*(수백 차원 좌표)로 바꿔, 가까운 좌표=비슷한 뜻인 조각을 코사인 유사도로 찾음 → 한국어 질문으로 영어 교과서도 검색(다국어 임베딩).
  - **sparse(BM25)**: *정확한 단어* 일치로 찾음(약물명·전문용어). 로컬 계산, API 없음.
  - **RRF(순위 융합)**: 두 결과의 순위를 역수로 합쳐 종합 top-k. 한쪽만 쓸 때보다 강건.
- **왜**: 의미검색(임베딩)은 동의어·번역에 강하고, 단어검색(BM25)은 정확 용어에 강함 → 상호보완.
- **디펜스**: "임베딩=의미 검색, BM25=용어 검색, RRF로 융합해 베끼지 않고 교과서 근거 위에서 판단합니다." *(임베딩이 뭐냐 추궁 시: "글을 수백 차원 좌표로 바꿔 가까운 좌표=비슷한 뜻. 표현이 달라도 의미로 찾습니다.")*
- **🔍 RAG 깊이 이해 (자주 묻는 것):**
  - **로컬 vs API**: *API 호출*은 ① 색인 때 교과서 조각을 벡터로 변환(임베딩 API) ② 검색 때 질문을 벡터로 변환 ③ 최종 생성(LLM API) — 이 셋뿐. *로컬*은 벡터 저장(`.npy` 파일)·BM25 계산·RRF 융합·코사인 유사도 계산. **서버가 필요 없다**(코퍼스가 작아서). "로컬로 돌렸다"=검색 파이프라인이 로컬, 임베딩·생성만 외부 API.
  - **코퍼스를 어떻게 참고하나(흐름)**: 질문 → (임베딩+BM25)로 코퍼스에서 관련 조각 top-k 검색 → 그 조각을 프롬프트에 `[근거1]…[근거3]`로 끼워넣음 → LLM이 그 근거를 *읽고* 답. = 시험 때 교과서 펴놓고 푸는 **오픈북**.
  - **코퍼스를 넣으면 왜 좋나(★핵심 구분)**: LLM 자체(가중치)는 *안 바뀐다* — RAG는 **학습/파인튜닝이 아니라, 추론할 때 참고자료를 그 순간 펴주는 것**. 효과 = ① 환각↓(지어내는 대신 근거 인용) ② 특정 도메인·최신 지식 주입 ③ 근거 추적 가능(어느 교과서 어느 부분) ④ 코퍼스 교체로 우리가 통제. "LLM이 똑똑해진다"기보다 *그 답변이* 근거 기반이라 신뢰↑.
  - **디펜스**: "RAG는 모델을 학습시키는 게 아니라 답할 때 신뢰 출처를 펴 보게 하는 오픈북입니다 — 그래서 환각이 줄고 근거를 댈 수 있고, 모델 가중치는 안 건드립니다(재학습 안 함 = 데이터 안전과도 직결)."
  - **⚠️ RAG 한계(정직, Codex 반영)**: ① "한국어 질문→영어 교과서"는 *원리상 가능*이지 성능 보장 아님(retrieval recall@k 평가 후속). ② BM25는 cross-lingual(한↔영) 못 함 — 영어 용어 매칭 보조일 뿐. ③ 검색이 엉뚱한 조각을 주면 오히려 환각↑(retrieval error도 failure mode) → near-copy 별도 + 평가셋으로 점검. ④ 현재 색인=Harrison 샘플뿐 → "교과서 근거"는 *제한된 내과 샘플 근거*로 좁혀 말할 것. ⑤ ⚠️ **저작권 교과서 청크를 외부 LLM(gpt-5.5)에 전송**하는 것은 거버넌스의 "원문 외부전송 금지"와 충돌하는 **미해결 이슈** — 프로덕션 전 *오픈라이선스 코퍼스 / 로컬 LLM / 비저작권 요약* 중 하나로 닫아야 함(no-training만으론 불충분).

### M3. ②A 구조 + ②B 임상 — 왜 검토를 둘로 나눴나
- **무엇**: ②검토가 두 층 — ②A는 형식·구조(체크리스트 항목 수·복합질문·채점 명료), ②B는 임상 내용(병력 누락·red flag·내적 논리), RAG 근거 위에서.
- **원리**: 한 프롬프트가 형식·내용을 동시에 보면 둘 다 얕아짐 → 역할 분리로 각각 깊게.
- **왜(증거 기반)**: 파일럿에서 구조만 보는 ②는 사례특이 임상 지적을 놓침(recall 23%) → 임상 리뷰어(②B) 추가로 +6%p. *구성 불일치를 데이터로 발견해 고친 것.*
- **디펜스**: "구조와 임상은 다른 전문성이라 분리했고, 그 필요성을 파일럿 데이터로 확인했습니다."

### M4. 생성 루프(LangGraph) — 어떻게 스스로 고치나
- **무엇**: ①생성 → ②심사 → (미흡하면)수정 → 재심사, 자동 반복.
- **원리**: LangGraph = *상태*(현재 사례·심사결과)를 들고 *노드*(생성·심사·수정)를 도는 그래프. **분기**(심사 결과로 종료 or 수정)와 **루프**(수정→재심사)를 선언적으로 표현.
- **종료조건**: ②A가 Accept/Minor *그리고* ②B must_fix=0일 때만 통과. 아니면 수정 후 다시 심사(최대 N회). 끝에 통과/예산소진을 명시 기록.
- **왜 LangGraph(함수 호출 대비)**: 상태·분기·루프·**추적·중단/재개**를 선언적으로 관리. 단순 함수 루프(`generator.generate`)도 병존하지만, 복잡한 분기·재개·추적엔 그래프가 유리. ⚠️ 한 번 생성보다 비평→수정 반복이 품질↑(멀티에이전트>단일, 선행연구 88% vs 31%). H2 안전성 확인 후 프로덕션(현재 프로토타입).
- **🔍 LangSmith(추적·관측)**: 각 노드의 입출력·토큰·지연·실패를 기록해 "왜 이 결과가 나왔나"를 디버깅·재현. **현재 미통합(코드 docstring 언급만) → 도입 예정.** ⚠️ 단 LangSmith는 SaaS(데이터 외부 전송) → **학생·사례 데이터는 비식별·로컬 우선 또는 self-host**(거버넌스 §M12와 충돌 주의).
- **디펜스**: "LangGraph로 생성-비평-수정 루프(상태·분기·재개)를 돌리고 구조·임상 둘 다 통과해야 멈춥니다. LangSmith 추적은 도입 예정이되 데이터 거버넌스(비식별·로컬)를 지킵니다. 최종 확정은 교수(HITL)."
- **🛡 LangGraph 집중 디펜스 (예상 공격 전부):**
  - *"어디에 썼나?"* → `graph.py` 한 곳, **①생성→②심사→수정→재심사 루프**. ③④ 학생 런타임은 *상호작용*이라 이 개발 그래프와 별개(`demo_pipeline`).
  - *"왜 LangGraph? 함수로도 되잖아"* → **솔직히 현재 단일 루프는 함수로도 충분**(그래서 `generator.generate` 함수 버전을 *병존*시킴). LangGraph의 값어치는 ① 상태 공유·조건부 분기·루프를 **선언적으로**(코드가 곧 흐름도) ② **체크포인트(중단/재개)** ③ **LangSmith 추적 연동** ④ 멀티에이전트 확장 — *지금의 필수가 아니라 확장·표준화 대비*.
  - *"오버엔지니어링 아닌가?"* → 인정하고 **핵심 경로는 함수도 유지**(그래프 강제 X). 분기·재개·추적이 늘면 그래프가 유지보수에 유리 — 의도적 layering.
  - *"함수랑 결과가 다른가?"* → 종료조건 동일(②A Accept/Minor · ②B must_fix=0). 그래프는 상태·로그·재개를 더 명시적으로 남길 뿐.
  - **한 줄 디펜스**: "단일 루프엔 함수로도 충분하지만, 분기·중단재개·추적·멀티에이전트 확장을 표준 골격으로 대비하려고 ①생성 루프에 LangGraph를 썼습니다. 과하지 않게 핵심 경로는 함수 버전도 병존합니다."

### M5. 블라인드 비교 — 왜 이게 공정한 검증인가
- **무엇**: 교수에게 같은 사례의 두 리뷰(교수 과거 피드백 vs AI)를 *어느 게 AI인지 숨기고* 무작위로 제시, 1~5 평가.
- **원리(편향 제거)**: AI인 줄 알면 점수가 끌려감(인지 편향) → 가려야 *내용의 질*만 평가. + 형식을 불릿으로 통일해 문체로 들키는 것 방지. + "어느 게 AI 같나"를 물어 **블라인드 성공률**을 점검(50% 근처 = 잘 가려짐).
- **왜**: "AI가 교수에 필적하나"를 *공정하게* 재려면 정체를 숨겨야. (v1은 AI 답을 먼저 보여줘 앵커링 → Codex가 깨서 재설계.)
- **디펜스**: "정체를 숨긴 블라인드라 문체가 아닌 내용으로 평가되고, 블라인드 성공률로 그 가림이 유효했는지까지 확인합니다."

### M6. 평가지표 완전 정리 — 무엇을 어떻게 재나
> precision/recall만이 아니다. 지표를 **성능 / 신뢰도 / 불확실성 / 판정기준** 4묶음으로.

**A. 성능 지표 (AI가 얼마나 잘하나)**
- **recall(재현율)** = 교수 지적 中 AI가 잡은 비율. "놓친 게 적나".
- **precision(정밀도)** = AI 지적 中 타당한 비율. "헛소리가 적나".
- **F1** = recall·precision의 조화평균(둘의 균형 종합). ⚠️ 우리는 매칭을 사람이 adjudication 안 해 **근사 pseudo-F1**(완전 F1은 본검증).
- **harmful gate(안전)** = AI가 위험·유해하게 지적한 건수 → **0건이어야**(성능 아닌 안전 게이트).

**B. 신뢰도 지표 (평가 자체를 믿나 = 사람끼리 일치)**
- **ICC** = 연속 점수(루브릭 1~5) 평가자 일치도(분산분석 기반: *사례 간 진짜 차이* 대 *평가자 잡음*). **ICC(2,k)**=k명 평균점수 신뢰도 / **ICC(2,1)**=단일 평가자 일반화. Likert는 순서형이라 ICC는 근사 → **weighted kappa 병행**.
- **kappa(카파)** = 범주 라벨 일치도, **우연 일치 보정**(찍어도 맞는 확률 제외). Cohen(2인)·Fleiss(3인+)·weighted(순서형, caught/partial/missed 거리 사전지정)·**Gwet AC / Krippendorff α**(평가자수 변동·결측·라벨 쏠림에 강건 → Fleiss 부적합 시).
- **blind 성공률** = 교수가 "어느 쪽이 AI"를 맞힌 비율. **50% 근처 = 잘 가려짐**(못 맞힘이 좋은 것). forced-choice라 **binomial 검정/CI 필요**(소표본선 67%도 우연 가능).

**C. 불확실성·강건성**
- **bootstrap 95% CI** = 재표집(수천 번) 흔들림 범위. **case-cluster** = 사례 단위 재표집(항목은 사례에 nested → 사례째 묶어 *가짜 정밀* 방지). ⚠️ n=6 파일럿선 **기술적 불확실성(descriptive)일 뿐** — 신뢰구간 강해석 금지.
- **LOO(leave-one-out)** = 사례 하나씩 빼고 재계산 → **민감도/강건성 분석**(한 사례에 휘둘리나).

**D. 판정 기준 (본검증)**
- **content validity** = 전문가가 사례·항목 타당성 판단. **비열등성 margin δ** = "AI가 교수에 *필적*"의 기준선(차이 CI 하한 > −δ), 교수 합의로 **사전 지정**.
- **⚠️ primary endpoint 1개 고정**: 본검증 전 주지표를 하나로(후보=블라인드 루브릭 차이). 지금은 미확정 — 여러 지표 병기는 탐색적.

**왜 이렇게 많나(디펜스)**: 성능(A)만 보면 과장 → 평가가 신뢰되나(B, 사람끼리 일치)부터 + 소표본 불확실성(C)까지 봐야 정직. **순서: 사람-사람 일치도(B floor) → 성능(A) → 강건성(C) → 본검증 판정(D).**

### M7. 거버넌스 — 베낌·누수를 어떻게 막나
- **near-copy 탐지**: 생성 사례를 짧은 창으로 잘라 원본(교과서·사례)에 *그 조각이 통째로 있나* 검사(문자 n-gram containment). 부분 복제도 잡음.
- **provenance·누수 분할**: 생성물에 출처·모델·버전 스탬프. 데이터를 train/dev/locked로 *미리* 나눠 평가셋 오염 방지.
- **왜**: 베끼면 저작권·독창성 문제, 같은 데이터를 생성·평가에 같이 쓰면 결과 무효 → 사전 차단.
- **디펜스**: "생성물이 원본을 베꼈는지 자동 검사하고, 평가 데이터는 잠가서 누수를 막습니다."

### M8. 검증 통계 설계 — 결과를 어떻게 "통과/실패" 판정하나
- **무엇**: 데이터가 나오면 통과/실패를 *사전 규칙*으로 판정(연구자 자의 차단).
- **요소**: **primary endpoint**(주지표 1개 고정) · 표본수/검정력 · 비열등성 margin δ · 다중비교 보정 · 결측 처리 · 평가자·사례를 *랜덤효과*로 보는 혼합모형.
- **현재 상태(정직)**: 파일럿이라 **미확정** — 본검증 전 OSF 사전등록. 지금은 기술통계(descriptive)만, 통과/실패 결론 없음.
- **디펜스**: "파일럿은 탐색적이고, 통과 기준·표본수·δ·주지표는 본검증에서 사전등록합니다. 지금 수치로 결론 내지 않습니다."

### M9. RAG 세부 파라미터 — 검색을 어떻게 조정하나
- **청킹**: 교과서를 ~700자 문단 조각으로. 현재 색인 = Harrison 샘플 1,200청크(전체 18권 아님).
- **검색**: dense top-n=12 + sparse top-n=12 → RRF(상수 60) → **top-k=3**만 프롬프트 주입.
- **임베딩**: `gemini-embedding-001`(다국어), 차원=모델 기본값. **컨텍스트 한계**: 사례 JSON `[:30000자]` truncation(토큰 한계 대비).
- **실패 모드(정직)**: 검색이 엉뚱한 조각을 주면(retrieval error) 오히려 해가 될 수 있음 → near-copy 별도 검사 + **retrieval recall@k 평가셋은 후속(B2)**.
- **디펜스**: "700자 청크·top-k=3·RRF 융합. 검색 실패도 failure mode로 인지하고 retrieval 평가셋을 따로 만듭니다."

### M10. LLM 재현성·비결정성 — 같은 입력이면 같은 결과인가
- **문제**: LLM은 확률적 → 같은 프롬프트도 답이 달라질 수 있음(비결정).
- **통제**: 추론 모델(gpt-5.5)은 temperature 미전송(기본), 보조콜 분리. **재현성 = 모델 snapshot·prompt version·코드 version을 provenance에 기록**(`governance.py`). JSON 파싱 실패·재시도 처리.
- **현재 상태**: 정확한 model id·파라미터를 로그/provenance에 고정, **검증 전 버전 freeze**.
- **디펜스**: "LLM은 비결정적이라 모델·프롬프트·코드 버전을 산출물에 박아 재현을 추적하고, 검증 직전 버전을 freeze합니다."

### M11. 구조화 출력(structured output) — AI 답을 어떻게 집계 가능하게 하나
- **무엇**: AI 산출(리뷰·채점)을 자유문이 아니라 **고정 JSON 스키마**(Pydantic)로 강제 — 카테고리(enum)·근거·수정안·심각도 필수 필드.
- **원리**: 자유문은 집계·검증 불가 → 계약된 구조라야 기계가 recall/precision을 계산. invalid 출력은 재시도.
- **안전**: 스키마 위반·prompt injection 방어, 의미 검증기(타임라인 일관성·불가능 vitals·남성 산부인과력 등).
- **디펜스**: "AI 출력을 JSON 스키마로 계약해 enum·필수필드를 강제하고 위반 시 재시도합니다 — 그래야 자유문이 아니라 집계 가능한 데이터가 됩니다."

### M12. 비식별·데이터 흐름 — 개인정보를 어떻게 지키나
- **비식별**: HWP 원문 → 환자·학생·기관·개발자명·전화·이메일·주민번호 제거(ingest deid), 전수 재검(PII 0 확인).
- **데이터 흐름**: 원본(`raw_private`, 로컬·gitignore) → 비식별 작업본 → train/dev/locked 분할. 외부 API엔 비식별만, 재학습 안 함.
- **API 데이터(정직)**: OpenAI/Google no-training tier(Google은 **유료 tier 필수** — free quota=학습), feedback 제출 금지. 학생 데이터(③④ 런타임) PIPA/IRB는 교수팀 영역.
- **디펜스**: "원문은 로컬·잠금, 외부엔 비식별만·재학습 안 함, 평가셋은 분할로 잠가 누수를 막습니다. 학생 데이터 IRB는 교수팀."

### M13. 가상환자 답변정책(③) — 어떻게 일관되게 연기하나
- **무엇**: ③가상환자가 사례 속 환자를 연기하며 학생 질문에 답.
- **원리(답변 정책)**: 항목별 *허용 사실* + 트리거(언제 답)·거부/되묻기(모르는 것)·**과공개 억제**(묻지 않은 정보 안 흘림)·일관성(모순 답변 방지)·**위험 의료조언 거부**.
- **현재 상태(정직)**: 완전한 유한상태기계(FSM)는 **설계안**, 현재 구현은 **프롬프트 기반** 정책(허용사실+과공개 억제). 음성(STT/TTS)은 옵션·후속.
- **디펜스**: "환자가 묻는 것만 답하고(과공개 억제), 모르는 건 되묻고, 모순·위험조언을 막는 정책입니다. 완전 FSM은 설계안이고 현재는 프롬프트 기반임을 명시합니다."

## 11. 예상 질문 디펜스 (Q&A)

**Q. 이거 결국 ChatGPT한테 사례 만들라는 것 아닌가?**
A. 아닙니다. 단일 프롬프트는 *상당수가 수정 필요*하다는 관련 연구가 있고(정확 수치·출처는 §8 인용 주의), 우리는 생성→비평→수정 **멀티에이전트 루프 + 체크리스트 골격 + RAG 근거 + 교수 확정(HITL)** 구조입니다. 검증 하네스로 품질을 확인합니다.

**Q. AI가 만든 사례가 사람이 만든 것만큼 좋은가?**
A. **동등하다고 주장하지 않습니다.** "AI=인간 동등" 주장은 관련 문헌에서도 적대검증에 탈락했습니다. 우리 입장은 **"전문가 검토 후 사용가능"** 이고, 그 검토를 ②가 돕습니다.

**Q. 의학적으로 틀린 내용(hallucination)은 어떻게 막나?**
A. 베끼지 않고 **교과서 근거(하이브리드 RAG)**를 보게 하고, ②B 임상 리뷰어가 red flag·위험인자·내적 논리를 점검합니다. 외부 연구(MedTutor)는 이런 구조로 오류를 ~3.3% 수준까지 낮출 수 있다고 보고합니다 — 다만 **우리 시스템의 오류율은 아직 측정 전**이며, 최종 안전성은 교수 검토(HITL)로 담보합니다.

**Q. 왜 환자가 다 "65세 남성 흡연자"인가?**
A. 그 편향은 실측된 문제(남성 79~84% 과대)입니다. 그래서 생성 단계에서 **인구통계 다양성을 유도**합니다(현재는 프롬프트 지시 수준이고, 진단별 분포 샘플러는 후속 과제).

**Q. 채점을 왜 0/1(이분법)으로 하나, 너무 단순하지 않나?**
A. 병력·진찰은 이분법이 SOTA에서 강력하고(오픈모델도 90.5%로 GPT-4o급), 채점자 신뢰도가 높습니다. 단 **환자의사관계(PPI)는 이분법이 부적합**해 루브릭+인간 보조로 따로 다룹니다.

**Q. ②검토 AI 성능이 23%면 낮은 것 아닌가?**
A. 그 23%는 **성능 수치가 아니라 진단용 신호**입니다. 당시 ②(일반 구조 점검)와 교수 피드백(사례특이 임상)의 **구성이 달랐고**(n=3, 자동 판정), 이를 발견해 ② 2층 분리로 개선했습니다(+6%p). 진짜 성능은 **교수 인간 판정 본검증**에서 측정합니다.

**Q. 신뢰도(일치도)는 어떻게 보고하나?**
A. **전문가 간 일치도(floor)를 먼저** 보고, 이분법 kappa·도메인별 weighted kappa·F1 + 부트스트랩 CI로 보고합니다. 정밀 ICC는 낮을 수 있어 주지표로 쓰지 않습니다.

**Q. 학생 데이터·개인정보는?**
A. **현재 학생 음성·대화는 미수집**입니다. 학생 런타임 도입 시, 외부 음성 API(STT/TTS)를 쓰기 전에 IRB·동의·비식별과 **로컬 STT 대체**를 검토하며, 민감정보의 외부 SaaS 추적은 금지합니다. 외부 API엔 비식별만 입력, 재학습 안 함.

**Q. 결제·모델이 계획서와 다른데?**
A. 계획서는 Claude였으나 결제 보류로 상위 GPT(gpt-5.5)로 대체했습니다. 어댑터 구조라 결제가 풀리면 한 줄로 Claude 복귀가 가능합니다. 모든 변경은 `transparency.md`에 정직히 기록합니다.

**Q. 상용 제품(메튜 등)과 뭐가 다른가?**
A. 우리는 **학술·오픈소스·무료·논문 목적**입니다. **공개: 코드·프롬프트·toy(가상) 데이터·평가 스크립트·집계 지표.** **비공개: 학교 사례·저작권 교과서**(원저작권 보호). 작동방식을 투명하게 공개하고 과대주장을 하지 않는 점이 상용과 다릅니다.

## 12. 부록

### 12.1 용어집
- **CPX/OSCE**: 표준화 환자로 학생 임상수행을 평가하는 실기시험.
- **표준화 환자(SP)**: 환자 역할을 일관되게 연기하도록 훈련된 사람(여기선 ③ AI가 대행).
- **RAG**: 검색 증강 생성 — 외부 근거를 찾아 AI 답변에 반영.
- **HITL**: Human-in-the-loop — 사람(교수)이 최종 확정.
- **recall(재현율)**: 교수 지적 중 AI가 잡은 비율. **precision(정밀도)**: AI 지적 중 타당한 비율.
- **kappa / ICC**: 평가자 간 일치도 지표. **blind(블라인드)**: 어느 쪽이 AI인지 모르게 하는 것.
- **PPI**: 환자의사관계(Patient-Physician Interaction), CPX 채점 영역.
- **harness(하네스)**: 시스템을 정답지와 대조해 검증하는 도구.
