## 0. 한눈에 보기 (요약)

CPX-AI는 **의대 임상수행평가(CPX) 교육을 돕는 AI 시스템**입니다. 한 문장으로 하면 **"CPX 사례 개발과 연습을 *보조*하는 연구용 AI 시뮬레이터"** 입니다(교수를 대체하지 않고 돕습니다). 시험 사례를 **쓰는 AI(①)**, 그것을 **검수하는 AI(②)**, 환자를 **연기하는 AI(③)**, 학생을 **채점하는 AI(④)** 네 명과, 이들을 받치는 뒷단(도서관·통역사·품질검사실)으로 이루어집니다.

현재는 **②(검수 AI)가 교수의 피드백을 얼마나 재현하는지**를 교수 블라인드 설문으로 검증하는 **파일럿 단계**입니다.

| 항목 | 내용 |
|---|---|
| **무엇** | CPX 사례 생성(①)·검토(②)·가상환자(③)·자동채점(④) + 검증 하네스 |
| **누가/왜** | 양산부산대병원 임선주 교수 연구. 부산대 학생 무료 제공·오픈소스·논문 목적 |
| **현재 단계** | ②검토 AI의 H2 검증 파일럿(웹 설문 배포·교수 평가 수집 중) |
| **핵심 원칙** | 투명성·재현성, **과대주장 금지**("AI=전문가 동등" 주장 안 함), 데이터 누수 방지 |
| **모델** | ①②=gpt-5.5, ③④=GPT-4o(미구현), RAG 임베딩=Gemini, 검색=BM25 |
| **솔직한 한계** | feasibility 파일럿(소표본). 타당성(통계적 동등) 주장 아님 |

> CPX(Clinical Performance Examination, 임상수행평가) = 의대생이 **표준화 환자(연기자)**를 직접 문진·진찰하고, 그 수행을 체크리스트로 평가하는 실기시험. 국가고시 항목.

## 1. 배경과 맥락 — 왜 이 프로젝트인가

### 1.1 문제
CPX 교육에는 두 가지 비용이 큽니다.
1. **사례 제작 부담** — 좋은 CPX 사례 하나를 만들려면 교수가 시나리오·체크리스트·환자 답변·채점기준을 모두 설계하고, 동료 검토를 거쳐야 합니다. 시간이 많이 듭니다.
2. **연습 기회 부족** — 학생이 표준화 환자와 연습하려면 사람(연기자)·공간·채점자가 필요해, 반복 연습이 어렵습니다.

### 1.2 접근
AI로 이 과정을 **보조**합니다. 기존 사례를 변형해 새 사례를 **생성(①)**하고, AI가 **검토(②)**해 교수 확정을 돕고, 확정된 사례로 **가상환자(③)**가 학생과 대화하고, **자동채점(④)**으로 즉시 피드백을 줍니다.

### 1.3 성격 (오해 방지 — 가장 중요)
- **학술 연구 프로젝트**입니다. 영리·사업·스타트업이 **아닙니다.**
- 목적은 **논문 + 검증(validation) + 투명성**입니다. 시장·가격·마케팅과 무관합니다.
- 산출물은 부산대 학생에게 **무료**, 작동방식은 **전면 공개(오픈소스)**, 결과로 **논문**이 따라옵니다(논문은 교수·연구진 주도, 김용하는 학부 개발 담당).
- 따라서 평가·비평은 **학술 엄밀성·검증 타당성·재현성·의학적 안전성·과대주장 방지** 축으로만 합니다.

### 1.4 개발·검토 방식
모든 핵심 작업은 **두 AI의 교차 검증**을 거칩니다. Claude가 설계·구현하고, **Codex(다른 AI 모델)가 적대적으로 검수**(red-team)합니다. 이 보고서 자체도 Codex 검수를 받습니다. 모든 작업은 `worklog.md`에 기록됩니다.

## 2. 전체 구조 (아키텍처 한 장)

![CPX-AI 작동방식](cpx-flow.png)

시스템은 **세 영역**으로 나뉩니다.

```
A. 사례 개발 루프 (교수 보조)        B. 검증 (현재 파일럿)         C. 학생 런타임 (목표)
  씨앗 사례                            교수 과거 피드백              확정 사례
    │                                      │ + ② AI 리뷰               │
    ▼                                      ▼                          ▼
  ① 생성 ──▶ ② 검토 ──▶ 확정?           블라인드 설문               ③ 가상환자
    ▲           │  (RAG 근거)            (교수 A/B 익명)             (GPT-4o)
    └─ 수정 ◀───┘                          ▼                          ▼
                ▼                        집계                       학생 대화
            확정 사례                  (recall·정밀도·ICC)            ▼
                                                                 ④ 자동채점 ─▶ 피드백
```

**등장 인물(비유로 이해하기):**

| 모듈 | 비유 | 하는 일 |
|---|---|---|
| ① `generator` | ✍️ 사례 작가 | 증상+진단(또는 기존 사례)으로 새 사례 초안 작성 |
| ② `reviewer` | 🔍 심사위원 | 점검표로 구조+임상 결함을 지적, 수정안 제시 (**현재 검증 대상**) |
| ③ `patient` | 🎭 환자 배우 | 사례 속 환자를 연기, 묻는 것만 답변(과공개 제어) |
| ④ `grader` | ✅ 채점관 | 학생 대화를 체크리스트와 대조해 점수·피드백 |
| `rag` | 📚 도서관 사서 | 질문에 맞는 의학 교과서 근거를 찾아 ②에게 제공 |
| `llm` | 🔌 통역사(어댑터) | 모든 AI 호출 창구. 모델 교체는 이 파일 한 곳만 수정 |
| `harness` | 🔬 품질검사실 | "이 채점·검토를 믿어도 되나"를 정답지와 대조해 숫자로 검증 |
| `graph` | 🏭 관리자(LangGraph) | 생성→검토→수정→재검토 자동 루프 관리 |

핵심 동사 3개로 기억: **만든다(①②) · 연습한다(③④) · 믿는다(검증 하네스).**

## 3. 작동 원리 (단계별 상세)

### 3.1 A. 사례 개발 루프

1. **입력**: 증상 + 진단, 또는 기존 CPX 사례(씨앗).
2. **① 생성**: AI 작가가 도서관(RAG)의 교과서 근거를 참고하며 사례 초안을 작성합니다. "65세 남성 흡연자"만 반복 생성되는 편향(→ §8)을 막기 위해 **인구통계 다양성을 유도**합니다. *(현재는 프롬프트 지시 수준이며, 진단별 분포 샘플러·검증기는 후속 과제입니다.)*
3. **② 검토**: 심사위원 AI가 두 층으로 점검합니다.
   - **②A 구조 리뷰어**: 체크리스트 항목 수, 복합질문 분리, 채점 명료성, 진찰·교육 항목 등 *형식·구조*.
   - **②B 임상 리뷰어**: 임상 타당성, 핵심 병력 누락, red flag, 위험인자, 내적 논리, 표준화 환자 현실성 등 *내용*. 이때 RAG 근거를 활용합니다.
   - 출력: 각 지적에 **카테고리·근거·수정안·심각도**를 붙입니다.
4. **확정 판단**: Accept / Minor / Major / Reject. 미흡하면 ①로 돌아가 수정 후 재검토(이 루프를 LangGraph가 자동화). **최종 확정은 교수(HITL, Human-in-the-loop)** 가 합니다.

### 3.2 B. 검증 (H2, 현재 파일럿)

"②의 AI 검토가 교수 피드백을 얼마나 재현하는가"를 측정합니다. 자세한 설계는 §7.

### 3.3 C. 학생 런타임 (목표 산출물)

1. **③ 가상환자**: 확정 사례로 환자를 연기합니다. **항목별 답변 정책** — 허용 사실, 트리거 표현, 거부·되묻기 규칙, **과공개 억제**(묻지도 않은 정보를 흘리지 않게). *(완전한 유한상태기계(FSM)는 설계안이며, 현재 구현은 프롬프트 기반입니다.)* 텍스트 우선, 음성(STT/TTS)은 옵션·후반.
2. **학생 대화**: 학생이 문진·교육을 진행, 대화록이 남습니다.
3. **④ 자동채점**: 대화록을 체크리스트와 대조합니다.
   - **병력청취·신체진찰 = 이분법**(도달/누락). 복합질문은 "또는"으로 분리.
   - **환자의사관계(PPI, 20%) = 이분법 부적합** → 루브릭 + 인간 보조.
   - 뜻이 같으면 표현이 달라도 인정(의미 기반). 키워드 방식(v0)도 병존해 비교.
4. **디브리핑(첨삭)**: 발화별 유도성 질문·전문용어·개방형/폐쇄형 여부 + "이렇게 말하라"는 리라이트 제안.

### 4.1 데이터 누수 방지 (연구 무효화 1순위 함정)
같은 사례를 *생성 예시 + 튜닝 + 가상환자 조정 + 최종평가*에 동시에 쓰면 결과가 조용히 무효가 됩니다. 그래서 데이터를 **train(생성용) / dev(튜닝용) / locked(평가 전용)** 로 **미리** 분리하고, locked는 평가 당일까지 열지 않습니다. 생성 사례와 실제 사례는 provenance 태그로 구분합니다. *(분할 manifest·해시·각 split 개수는 `data/working/`에 기록되어 "locked를 안 열었다"를 사후 검증 가능.)*

### 4.2 보안·저작권
- 학교자산(사례·피드백)과 저작권 교과서는 **git 커밋 금지.**
- 외부 AI API에는 **비식별 데이터만** 입력하고, **재학습(파인튜닝)하지 않습니다**(프롬프트 + RAG 방식). 외부 API는 **no-training(학습 미사용) 설정**으로 사용합니다(OpenAI·Google).
- 키는 `.env`(비공개). 배포 설문은 비식별 항목만, 암호 게이트 + 검색엔진 차단.

## 6. RAG (검색 증강 생성) 작동방식

RAG = AI가 **"오픈북 시험"** 처럼 교과서 근거를 보고 검토하게 하는 구조입니다. 베끼지 않고 근거를 대게 합니다. (`src/cpx/rag.py`)

1. **색인(1회)**: 교과서를 조각(청크)으로 나눠 → 임베딩 API(Gemini)로 각 조각을 벡터로 변환 → **로컬 파일**(.npy)에 저장. *(현재 인덱스 = Harrison 1권 1,200청크 샘플. 18권 전체는 코퍼스 후보이며 아직 색인 전입니다.)*
2. **검색(질의 시, 하이브리드)**:
   - **dense**: 질문을 벡터로 바꿔 의미가 비슷한 조각을 코사인 유사도로 찾음(한국어 질문 → 영어 책도 가능).
   - **sparse**: BM25로 정확한 용어가 일치하는 조각을 찾음(로컬 계산).
   - **RRF 병합**: 두 결과의 순위를 합쳐 최종 top-k.
3. **주입+생성**: top-k 근거를 ② 프롬프트에 넣어, gpt-5.5가 근거 기반으로 검토.

- **외부 API 호출 = 색인 시 청크 임베딩(1회) + 검색 시 질의 임베딩(매 질의) + 생성 LLM 호출.** 벡터 저장은 파일, BM25·병합은 메모리 → **벡터DB 서버가 필요 없습니다**(코퍼스가 작아서). 수백만 청크 규모면 벡터DB가 필요하지만, 우리 규모는 아닙니다.
- **현재 한계(중요)**: 프로젝트 원칙은 *한국 CPX 자료를 주 코퍼스*로 두는 것이지만, **현재 색인된 것은 영어 교과서(Harrison 샘플)뿐**입니다. 한국어 질의는 dense(다국어)가 주력이며, 한국 CPX 총론·지침·점검표 색인과 저작권 처리는 **후속 과제(gap)**입니다.

## 7. 검증(H2) 설계 — 핵심

> **H2 가설**: "AI ②검토자가 전문가(교수) 피드백을 재현/필적하는가?"

### 7.1 왜 설계가 까다로운가 (v1의 결함, Codex가 깸)
처음 설계(v1)는 "AI가 교수 지적을 쪼개고 → AI가 매칭하고 → 교수가 맞나 확인"이었습니다. Codex가 다음을 지적해 **REVISE**:
1. **순환 측정** — AI가 만든 후보 안에서만 교수가 판정 → 결과가 모델 산물인지 교수 판정인지 분리 안 됨.
2. **앵커링** — AI 답을 먼저 보여주면 교수가 끌려감(인지 편향).
3. **정보 부족** — 1줄 지적으로는 판정 불가 → 평가자 간 신뢰도 붕괴.
4. **약한 주지표** — 항목별 recall은 부차적. 주지표는 사례 단위 비교여야.
5. **정밀도 누락** — AI가 새로 낸 지적의 타당·유해성 평가 필요.

### 7.2 v2/v3 재설계 (현재 배포본)
**판정 단위 = 사례. 한 사례 = 화면에 전부 표시**(초안 전문 + 두 리뷰 전문 + AI 리뷰 전체).

- **1단계 [주지표] 블라인드 루브릭** — 같은 초안의 두 리뷰(하나는 교수 피드백, 하나는 AI)를 **무작위·익명**으로 제시. 교수가 어느 쪽이 AI인지 모른 채 **완전성·정확성·유용성·안전성(1~5)** 로 평가. → AI 리뷰와 전문가 피드백의 **탐색적 유사도·차이**를 측정(앵커링 제거). *(현재는 동등성/비열등성을 주장하지 않으며, 비열등성 margin·표본수·primary endpoint는 본검증에서 사전등록합니다.)* 단, 같은 교수가 *자신이 과거에 쓴* 피드백을 평가할 경우 완전한 블라인드가 아닐 수 있어, 평가자별 **원 피드백 작성자 여부**를 기록해 분석에서 분리합니다.
- **2단계 [보조] recall** — 어느 쪽이 AI였는지 공개 후, AI 리뷰 전체를 보며 교수 지적 각각을 caught/partial/missed/excluded.
- **3단계 정밀도** — AI가 낸 지적 각각: 타당·중요 / 타당·경미 / 중복 / 틀림 / **위험·유해**.

**블라인드가 깨질 위험**(AI와 교수의 문체·길이 차이로 들킴)을 막기 위해 v3에서:
- 두 의견을 **같은 형식(불릿)으로 정규화** + 순서 무작위.
- **"어느 쪽이 AI 같나 + 확신도"** 를 물어 **블라인드 성공률**을 점검(50% 근처면 양호).
- 각 1~5 척도에 **BARS 앵커**(1/3/5 기준 서술) + 전문 **읽음 확인** 체크.
- **모드 분리**(`?mode=blind`/`recall`/`full`)로 평가자 코호트를 나눠 단계 간 오염 방지.

### 7.3 지표
- (1) 차원별 **AI − 전문가 차이 + 사례 클러스터 부트스트랩 95% CI**, **ICC(2,k)**.
- (2) recall(다수결 합의) + **Fleiss kappa**(현재 구현) + 카테고리별. *(weighted/Gwet kappa는 본검증 예정.)*
- (3) 정밀도 / false positive / **harmful = 안전 게이트**: 사전 정지규칙으로 운용(예: harmful 판정 1건 이상이면 릴리즈 보류).
- (4) 블라인드 성공률.
- 판정자 N명이 독립 평가 → 평가자 간 일치도 보고. **전문가-전문가 일치도(floor)를 먼저** 봅니다.

### 7.4 하이브리드 설문 (교수 부담 ↓ + 데이터 확보)
교수는 사례마다 **"블라인드 평가"(정량 데이터) 또는 "열람만"(자유 메모)** 를 고릅니다. 원하는 사례만, 언제든 종료 가능. 블라인드 무결성을 지키면서(평가를 라벨 보기 *전에* 함) 부담을 낮춥니다. 배포 = `web/adjudication/`(암호 게이트·중앙 수집·비식별). *(현재 집계 스크립트는 full/blind/recall 모드를 지원하며, 하이브리드 응답의 자동 집계는 데이터 수집 후 추가 예정입니다 — 그 전까지 하이브리드는 "권장"이 아니라 "선택지"로 안내합니다.)*

### 7.5 파일럿 결과 (솔직하게)
- dev_tune 3건, **자동 LLM 판정**(소표본, locked 미사용): **② recall ≈ 23%**. ⚠️ 이 수치는 성능 측정이 아니라 **프롬프트 디버깅용 신호**입니다(아래 개선 수치도 동일 한계).
- **Codex 판정**: 이 23%는 **두 신호의 혼합** —
  1. *구성 불일치 아티팩트*: 당시 ②는 일반 구조 점검표였는데 교수 피드백은 사례특이 임상 정교화라 결이 달랐음(n=3, 자동 판정, 교수 피드백이 순수 정답지가 아님).
  2. *②의 진짜 한계*: 사례특이 임상 비평이 약함.
- **조치**: ② 2층 분리(②A 구조 + ②B 임상) + 실험 재설계(인간 adjudication). ②B 추가 후 19% → 26%(자동 판정·소표본이라 역시 **신호 수준**, 성능 주장 아님).
- **결론**: 본검증 = 교수 인간 판정 + 카테고리별 + ②B 반복 개선. **과대주장 금지.**

## 8. 학술적 근거와 선행연구

> 출처: `docs/research-llm-cpx-sota.md` (134개 주장 → 25개 적대검증 → **22 확인 / 3 폐기**). 아래는 **확인된 것만** 근거로 씁니다.

### 8.1 단계별 SOTA
- **① 생성**: 구조화된 **멀티에이전트 프롬프팅 > 단일 RAG**. 멀티에이전트로 사용가능 OSCE 88% vs 단일 RAG는 31% 수정 필요(Wang 2024, JMIR 10.2196/82116 — *영어 OSCE 사례·전문가 사용가능성 평가 기준이며, 한국 CPX와 과제·지표가 동일하진 않음*). **"checklist-as-script"**(체크리스트를 생성 골격으로) 기법 확인 → 우리가 `[붙임2]` 양식을 생성 골격으로 쓰는 근거.
- **② 검증**: 지배적 방법 = 다수 전문가 내용타당도. **순서 = 전문가 간 일치도 먼저 → AI-전문가**. 전문가 HITL 수용률 ~87%. 단 정밀 품질 ICC는 낮을 수 있음(0.18) → 이분법 kappa 위주. silver-label로 부트스트랩.
- **③ 가상환자**: 학술 근거가 **가장 빈약 = 우리 기여 기회**. 차별점 후보 = 응답 일관성·정보 과공개 제어.
- **④ 채점**: **이분법 채점이 강력**(외부 벤치마크 기준). 한 평가에서 오픈모델(qwen3-32b)이 90.50%로 GPT-4o와 동급 — 단 *그 벤치마크 과제 기준*이며 한국 CPX 대화 채점과 동일하지 않습니다. 우리 H4-real 검증 전에는 **비용 절감 후보**로만 봅니다. 우리 D1(이분법) 결정과 방향 일치.

### 8.2 우리 방법의 위치
멀티에이전트+checklist-seed, 하이브리드 RAG, HITL 2라운드 심사 = **SOTA 정렬(차용)**. 블라인드/엄밀 검증, 가상환자 일관성·과공개 제어 = **차별화·기여 포인트**.

### 8.3 알려진 함정과 대응
- **인구통계 편향**("65세 남성 흡연자" 과생성): 실측으로 남성 79~84% 과대, 소수집단 0~1.5% 과소(PMID 40665345). → **생성 단계에서 인구통계 다양성 강제**(진단조건부 분포 샘플링).
- **의학 hallucination**: 하이브리드 RAG + 구조화 self-check가 효과적이라는 **외부 연구 보고**(MedTutor, 오류 ~3.3%). 이는 *문헌 수치*이며 **우리 시스템의 오류율은 아직 측정 전**입니다(현재 RAG는 Harrison 샘플이라 더 보수적으로 봐야 함). → 우리 RAG 설계의 *방향* 근거.
- **지표**: 정밀 ICC는 낮을 수 있으니 **이분법 kappa·도메인별 weighted kappa·F1 + bootstrap CI**, 인간-인간 floor 먼저.

> ⚠️ **인용 금지(적대검증 탈락)**: "AI 생성 문항이 인간과 심리측정적으로 동등"(npj) 등은 검증에서 폐기됨 → **우리 입장은 "전문가 검토 후 사용가능"**, "AI=인간 동등"은 주장하지 않음.

## 9. 한계와 정직성 (과대주장 방지)

- 현재는 **엔지니어링 smoke + feasibility 파일럿** 단계입니다. **타당성(통계적 동등) 주장이 아닙니다.**
- H2 파일럿은 사례 6건 수준 — "AI가 전문가에 필적/대체한다"고 **주장하지 않습니다.**
- 인간-AI 산문 블라인드는 문체 차이로 완전하지 않습니다(문헌도 인정) → 한계로 명기.
- 정답지(교수 피드백) 분해가 아직 일부 LLM 보조 → 향후 인간 codebook 고정 권장.
- 사전등록(OSF)·비열등성 margin·locked 최종평가는 **본검증 단계**에서.
- ③④ 학생 런타임은 코어는 있으나 음성·통합·실검증(H4-real: 실제 학생 대화 + 임상교원 라벨)은 미완.
- **IRB/윤리**: 현재 학생 데이터는 **미수집**입니다. H2 교수 설문은 비식별·자발적이며 IRB 필요 여부/면제는 교수팀이 판단합니다. 향후 **학생 런타임(③④)은 IRB·동의·PIPA 절차가 필요**하며 이는 교수팀 영역입니다.

## 10. 로드맵 (남은 일)

> 목표 = 검증된·투명한·오픈소스 CPX AI(부산대 학생 무료) + 결과로 따라오는 논문. 빌드-주도.

**바로 다음 (게이트):** 교수 파일럿 실행 → 집계 → **첫 H2 실결과**. 이후:
1. 결과로 **②B 보강** → 생성→검토→수정 루프로 **① 생성 품질** 향상("개선봇이 핵심").
2. **④채점 검증(H4-real)** — 실제 학생 대화 + 임상교원 라벨 → AI-인간 일치도(논문 핵심).
3. **지표·재현성·오픈소스 패키징·투명성 산출물**(논문 재료).
4. **본검증** — 사례·판정자 확대, 사전등록(OSF), 비열등성 δ.

**작은 작업(코드):** 하이브리드 응답 집계 지원, F1 지표 추가, RAG 코퍼스 확대(18권 전체·한국어 색인), 음성 완성도.

## 11. 예상 질문 디펜스 (Q&A)

**Q. 이거 결국 ChatGPT한테 사례 만들라는 것 아닌가?**
A. 아닙니다. 단일 프롬프트는 31%가 수정 필요(Wang 2024)인 반면, 생성→비평→수정 **멀티에이전트 루프 + 체크리스트 골격 + RAG 근거 + 교수 확정(HITL)** 구조입니다. 검증 하네스로 품질을 숫자로 확인합니다.

**Q. AI가 만든 사례가 사람이 만든 것만큼 좋은가?**
A. **동등하다고 주장하지 않습니다.** "AI=인간 동등" 주장은 관련 문헌에서도 적대검증에 탈락했습니다. 우리 입장은 **"전문가 검토 후 사용가능"** 이고, 그 검토를 ②가 돕습니다.

**Q. 의학적으로 틀린 내용(hallucination)은 어떻게 막나?**
A. 베끼지 않고 **교과서 근거(하이브리드 RAG)**를 보게 하고, ②B 임상 리뷰어가 red flag·위험인자·내적 논리를 점검합니다. 외부 연구(MedTutor)는 이런 구조로 오류를 ~3.3%까지 낮췄다고 보고합니다 — 다만 **우리 시스템의 오류율은 아직 측정 전**이며, 최종 안전성은 교수 검토(HITL)로 담보합니다.

**Q. 왜 환자가 다 "65세 남성 흡연자"인가?**
A. 그 편향은 실측된 문제(남성 79~84% 과대)입니다. 그래서 생성 단계에서 **진단조건부 인구통계 분포를 강제**해 다양성을 확보합니다.

**Q. 채점을 왜 0/1(이분법)으로 하나, 너무 단순하지 않나?**
A. 병력·진찰은 이분법이 SOTA에서 강력하고(오픈모델도 90.5%로 GPT-4o급), 채점자 신뢰도가 높습니다. 단 **환자의사관계(PPI)는 이분법이 부적합**해 루브릭+인간 보조로 따로 다룹니다.

**Q. ②검토 AI 성능이 23%면 낮은 것 아닌가?**
A. 그 23%는 **성능 수치가 아니라 진단용 신호**입니다. 당시 ②(일반 구조 점검)와 교수 피드백(사례특이 임상)의 **구성이 달랐고**(n=3, 자동 판정), 이를 발견해 ② 2층 분리로 개선했습니다(+6%p). 진짜 성능은 **교수 인간 판정 본검증**에서 측정합니다.

**Q. 신뢰도(일치도)는 어떻게 보고하나?**
A. **전문가 간 일치도(floor)를 먼저** 보고, 이분법 kappa·도메인별 weighted kappa·F1 + 부트스트랩 CI로 보고합니다. 정밀 ICC는 낮을 수 있어 주지표로 쓰지 않습니다.

**Q. 학생 데이터·개인정보는?**
A. **현재 학생 음성·대화는 미수집**입니다. 학생 런타임 도입 시, 외부 음성 API(STT/TTS)를 쓰기 전에 IRB·동의·비식별과 **로컬 STT 대체**를 검토하며, 민감정보의 외부 SaaS 추적은 금지합니다. 외부 API엔 비식별만 입력, 재학습 안 함.

**Q. 결제·모델이 계획서와 다른데?**
A. 계획서는 Claude였으나 결제 보류로 최상위 GPT(gpt-5.5)로 대체했습니다. 어댑터 구조라 결제가 풀리면 한 줄로 Claude 복귀가 가능합니다. 모든 변경은 `transparency.md`에 정직히 기록합니다.

**Q. 상용 제품(메튜 등)과 뭐가 다른가?**
A. 우리는 **학술·오픈소스·무료·논문 목적**입니다. **공개: 코드·프롬프트·toy(가상) 데이터·평가 스크립트·집계 지표.** **비공개: 학교 사례·저작권 교과서**(원저작권 보호). 작동방식을 투명하게 공개하고 과대주장을 하지 않는 점이 상용과 다릅니다.

## 12. 부록

### 12.1 용어집
- **CPX/OSCE**: 표준화 환자로 학생 임상수행을 평가하는 실기시험.
- **표준화 환자(SP)**: 환자 역할을 일관되게 연기하도록 훈련된 사람(여기선 ③ AI가 대행).
- **RAG**: 검색 증강 생성 — 외부 근거를 찾아 AI 답변에 반영.
- **HITL**: Human-in-the-loop — 사람(교수)이 최종 확정.
- **recall(재현율)**: 교수 지적 중 AI가 잡은 비율. **precision(정밀도)**: AI 지적 중 타당한 비율.
- **kappa / ICC**: 평가자 간 일치도 지표. **blind(블라인드)**: 어느 쪽이 AI인지 모르게 하는 것.
- **PPI**: 환자의사관계(Patient-Physician Interaction), CPX 채점 영역.
- **harness(하네스)**: 시스템을 정답지와 대조해 검증하는 도구.
